# resignation-flow — Delta Spec

## MODIFIED Requirements

### Requirement: 离职生效处理

离职生效时系统 SHALL：将员工状态置为「已离职」、禁用其系统账号（真实持久化）、**将工号后缀序号回收到可复用池**、**从关联考勤组中移除该员工**、**标记员工薪资档案的核算截止日期为离职日期**、保留员工档案可查询。

#### Scenario: 离职生效禁用账号

- **WHEN** 员工离职生效
- **THEN** 其 `sys_user.status` 被更新为禁用并持久化，该账号无法登录

#### Scenario: 离职生效释放工号

- **WHEN** 员工离职生效
- **THEN** 系统将该员工工号的后缀序号标记为可复用，后续新员工入职时优先从可复用池中分配

#### Scenario: 离职生效释放考勤组与薪资截止

- **WHEN** 员工离职生效
- **THEN** 系统将其从考勤组移除，并标记薪资核算截止至离职日期（次月起不再核算）

### Requirement: 离职档案脱敏查询

系统 SHALL 保留离职员工档案可查询，但展示时 MUST 对敏感信息按以下规则脱敏：身份证号保留前 3 位和后 4 位（`110***********1234`）、手机号保留前 3 位和后 4 位（`138****5678`）、银行卡号保留后 4 位（`****1234`）、薪资字段不返回明细仅返回最后一个月的实发汇总。

#### Scenario: 查询离职员工档案脱敏

- **WHEN** 查询一名已离职员工的档案
- **THEN** 其身份证号以 `110***********1234` 格式返回，手机号以 `138****5678` 格式返回，银行卡号以 `****1234` 格式返回，薪资仅显示最后一月实发金额汇总

#### Scenario: HR 经授权查看完整离职档案

- **WHEN** HR 专员因合规审计需要通过专门的审计接口查询离职员工完整档案
- **THEN** 系统记录审计日志后返回脱敏前的原始数据
