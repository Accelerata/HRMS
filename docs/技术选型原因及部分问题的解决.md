# 技术选型原因及部分问题的解决

> 本文档记录了 HRMS「入转调离」全流程开发中的关键技术选型决策及其原因，并汇总了设计阶段提出的 Open Questions 的处理策略。

---

## 一、为什么不引入工作流引擎（Activiti / Flowable）

### 结论

**沿用现有模板驱动引擎 `ApprovalStateMachineService` 并增强，不引入工作流引擎。**

### 当前审批场景的实际复杂度

四大流程的审批链均为简单的串行模式：

| 流程 | 审批链 | 步骤数 |
|------|--------|:------:|
| 入职 | 部门负责人 → HR 专员 | 2 步串行 |
| 转正 | 部门负责人 → HR 专员 | 2 步串行 |
| 调岗 | 原部门负责人 → 新部门负责人 → HR 专员 | 3 步串行 |
| 离职 | 部门负责人 → HR 专员 | 2 步串行 |

**全部是简单串行链**，没有并行分支、条件网关、子流程、边界事件——这些才是工作流引擎真正发挥价值的场景。

### 当前引擎只需要三类小改动

现有 `ApprovalStateMachineService` 约 250 行，结构清晰。满足需求只需：

1. **顺序门控**（约 15 行）：查询是否有更低 `step_order` 的未处理记录
2. **超时字段 + 定时扫描**（约 30 行）：`ApprovalRecord` 加 `due_time`，定时任务扫超时记录
3. **审批通知回调**（约 10 行）：通过/拒绝/退回后调用 `NotificationService`

**改动总量不超过 100 行代码。**

### 引入工作流引擎的真实成本

| 维度 | 成本 |
|------|------|
| 依赖体量 | Activiti 7 引入 50+ 传递依赖，Flowable 类似 |
| 数据库 | 引擎自身需要 20+ 张表（ACT_* / FLW_*），与现有 schema 正交 |
| 学习曲线 | BPMN 2.0 规范、引擎 API、流程部署/版本管理——团队需要 1-2 周上手 |
| 迁移代价 | 四个流程的 Service 全部重写，现有 `ApprovalStateMachineService` 废弃 |
| 调试成本 | 出问题时要在引擎内部表和业务表之间交叉排查 |
| 过度设计 | 可视化配置、委托审批、复杂工作台均声明为 Non-Goal |

### 什么时候才该上工作流引擎

以下信号出现时再考虑引入：

- 审批流程需要**业务人员自助配置**（而非开发写 SQL 种子数据）
- 出现**并行会签**（如财务 + 法务同时审批，全部通过才进入下一步）
- 出现**条件分支**（根据金额大小走不同审批链）
- 需要**流程版本管理**（旧流程实例继续用旧版本跑完）
- 审批节点数 ≥ 5 级且频繁变动

> **一句话**：审批场景是 2-3 步的简单串行链，~250 行的模板引擎增补不到 100 行即可满足需求。引入 Activiti/Flowable 相当于用交响乐团演奏《小星星》——技术上可行，但毫无必要。

## 二、什么是"业务人员自助配置"

### 当前方式：开发写 SQL 种子数据

审批流程通过 `phase3-approval.sql` 中的 INSERT 语句定义：

```sql
INSERT INTO `approval_template` (`business_type`, `step_order`, `approver_target`, `step_name`) VALUES
(1, 1, 'dept_manager',     '目标部门主管审批'),
(1, 2, 'hr_specialist',     'HR专员审批'),
-- ...
```

修改审批流程需要：开发改 SQL → DBA 执行 → 重启服务 → 测试验证，走完整的发布流程。

### 引入工作流引擎后：HR 后台自助拖拽配置

HR 总监在后台页面上拖拖拽拽就能改审批流程，不需要开发人员参与。例如：
- 给某个部门加一个"财务审核"环节
- 薪资超过 2 万的调岗额外加一级审批
- 调整审批顺序

### 为什么当前不需要

入转调离是企业最稳定的流程——入职就是部门 + HR 两步，调岗就是原部门 + 新部门 + HR 三步。这些规则十年如一日，不需要 HR 天天改。需要自助配置的场景通常是报销审批、合同审批等复杂多变的流程，而 HRMS 当前阶段根本不需要这些。

## 三、为什么不需要 Quartz / xxl-job

### 当前定时任务清单

| 任务 | 频率 | 逻辑 |
|------|------|------|
| 试用期到期提醒 | 每天 08:00 一次 | 扫 employee 表，到期前 7 天发通知 |
| 离职到期自动过渡 | 每天 00:30 一次 | 扫 employee 表，待离职且到期则改状态 |
| 审批超时催办 | 每小时一次 | 扫 approval_record 表，超 48h 未处理发通知 |

### 能力对比

| 能力 | Spring `@Scheduled` | Quartz / xxl-job |
|------|:--:|:--:|
| 按 cron 定时执行 | ✅ | ✅ |
| 动态增删任务（不停机） | ❌ 需改代码重启 | ✅ 管理后台点一下 |
| 分布式调度（多实例只跑一次） | ❌ 多实例会重复跑 | ✅ |
| 任务分片（海量数据并行处理） | ❌ | ✅ |
| 失败重试 / 可视化监控 | ❌ | ✅ |
| 依赖体量 | 零（JDK 自带） | xxl-job: 调度中心 + 7 张表 |

### 为什么不需要这些高级能力

- **动态增删任务**：入转调离的定时提醒规则是固定的，不需要动态修改
- **分布式调度**：当前是单体 Spring Boot 部署，`@Scheduled` 天然只跑一次
- **任务分片**：一家公司的员工数据量，单条 SQL 毫秒级完成
- **失败重试**：三个任务的核心逻辑都是扫描 + 更新，失败概率极低；真失败了，下一次调度会自然覆盖

> **一句话**：Spring `@Scheduled` 对付这三个低频短耗时任务绰绰有余，Quartz/xxl-job 是分布式集群 + 海量任务 + 动态管理时才需要的重型武器。

## 四、不需要用 Redis 做定时

### Redis 做定时的两种方式

**方式 A：Key 过期通知（Keyspace Notification）**

```
SETEX "remind:employee:123:probation" 604800 "1"
→ 监听 __keyevent@0__:expired 事件触发提醒
```

不适合原因：过期通知是"尽力而为"的，key 过期后可能延迟数秒到数分钟才通知，极端情况直接丢失。且为每个员工设一个 key，管理成本高于一条 SQL。

**方式 B：Redis 分布式锁防止多实例重复跑**

```java
if (redis.setIfAbsent("lock:probation_remind", "1", 300, TimeUnit.SECONDS)) {
    try { doRemind(); } finally { redis.delete("lock:probation_remind"); }
}
```

当前不需要原因：单实例部署，`@Scheduled` 默认不并发跑同一任务。将来多实例部署后需要时再加这一行代码即可，不需要提前引入复杂度。

> **一句话**：定时逻辑全量扫描比逐条倒计时更可靠，单实例无需分布式锁。

## 五、当前实现是否影响后期功能扩展

### 委托审批（需求 8.3）

| 改动点 | 改动量 | 破坏现有代码？ |
|--------|--------|:--:|
| 新增 `approval_delegation` 表 | 一张新表 | ❌ 纯新增 |
| 新增 `DelegationService` | 一个新 Service | ❌ 纯新增 |
| `processApproval()` 审批人校验 | 加一个条件判断 | ✅ 向后兼容 |

当前校验逻辑：
```java
if (!record.getApproverId().equals(currentUserId)) {
    throw BaseException.badRequest("您不是该审批步骤的审批人");
}
```

加委托后只需扩展一个条件：
```java
boolean isApprover = record.getApproverId().equals(currentUserId);
boolean isDelegate = delegationService.isActiveDelegate(
    record.getApproverId(), currentUserId, record.getBusinessType());
if (!isApprover && !isDelegate) {
    throw BaseException.badRequest("您不是该审批步骤的审批人");
}
```

### 审批人工作台高级功能（需求 8.2 大部分）

高级功能包括：批量审批、流转图、审批效率统计、抄送。

- **批量审批**：新端点循环调 `processApproval()`，不动引擎
- **流转图**：纯查询端点，不动引擎
- **审批效率统计**：按审批人聚合查询，不动引擎
- **抄送/知会**：新表 + 调用已预留的 `NotificationService` 接口

**这些全部是"加新端点/新表"**，不需要动 `ApprovalStateMachineService` 核心逻辑。

### 不会大改的原因

1. **审批引擎和展示层已解耦**：`ApprovalStateMachineService`（引擎）与 `ApprovalWorkbenchService`（展示）独立
2. **VO 已预留字段**：`ApprovalTodoVO` 有 `applicantName`、`stepName` 字段可复用
3. **`NotificationService` 是天然扩展点**：抄送通知、委托通知都通过此接口发送
4. **数据库模型通用**：`approval_record` 结构已覆盖委托场景，加列是 nullable 的

> **一句话**：当前设计没有给后期功能挖坑。委托审批和高级工作台落在展示层和审批人解析层，不是核心逻辑层。加功能 = 加代码，不改旧代码。

## 六、NotificationService 通知抽象设计

### 当前状态

项目通知设施为零：无 RocketMQ 依赖、无 notification 相关类、无 notification 表。

### 设计方案

```
┌─────────────────────────────────────────────────────────┐
│  NotificationService 接口                                │
│                                                         │
│  ┌─────────────────────┐    ┌─────────────────────────┐ │
│  │ InAppNotification   │    │ RocketMqNotification    │ │
│  │ (本期 2-3 小时落地)  │    │ (后期 2-3 天落地)        │ │
│  │                     │    │                         │ │
│  │ • 写 notification   │    │ • 实现同一接口           │ │
│  │   表（站内信）       │    │ • 发 RocketMQ 消息       │ │
│  │ • 打日志            │    │ • Consumer 发邮件/企微   │ │
│  │                     │    │ • 失败重试、死信处理     │ │
│  └─────────────────────┘    └─────────────────────────┘ │
│                                                         │
│  切换方式：改一个 @ConditionalOnProperty 配置，调用方零改动  │
└─────────────────────────────────────────────────────────┘
```

### 为什么用接口抽象

调用方（审批引擎、定时任务、四个流程 Service）只依赖接口：

```java
@RequiredArgsConstructor
public class ApprovalStateMachineService {
    private final NotificationService notificationService;  // 接口，不是具体实现
}
```

切换实现时，调用方一行代码不用改——这是策略模式 / 依赖倒置的经典应用。

### 为什么本期用站内信落地

三期需求同时压上来，通知是辅助设施而非核心主流程。引入 RocketMQ 需要额外 2-3 天（搭建服务端、编写 Producer/Consumer、集成邮件、处理重试和死信），用站内信只需 2-3 小时（一张表 + 一个实现类）。站内信本身也有业务价值——员工登录后看到"入职申请已通过"，不需要邮件也能完成闭环。

### 为什么预留 RocketMQ 而非本期就做

CLAUDE.md 要求了 RocketMQ，但当前通知场景中除审批待办外，其余均对实时性要求不高。审批待办的即时性问题可以在后期通过 RocketMQ Consumer 发邮件/企微/钉钉推送来补充。用一个 2-3 小时的方案先让通知体系跑起来，用真实数据验证通知类型和内容模板，再花 2-3 天接 RocketMQ——顺序合理。

## 七、关键风险及化解策略

### 风险 1：改动面大

**策略**：分阶段执行，底座先行，再逐个流程，每阶段编译 + 测试通过再进入下一阶段。

```
Phase 1: 公共底座（枚举、工号生成器、账号服务、定时任务框架）
    ↓ 编译 + 单测通过
Phase 2: 审批引擎增强（顺序门控、超时催办）
    ↓ 编译 + 单测通过
Phase 3: 入职流程 → Phase 4: 转正 + 调岗 → Phase 5: 离职
    ↓ 每阶段独立可验证
Phase 7: 全量回归测试
```

每完成一个模块检查权限管理是否完成（CLAUDE.md 要求）。

### 风险 2：工号并发

**策略**：DB 唯一约束 + 捕获 `DuplicateKeyException` 重试。

```sql
ALTER TABLE employee ADD UNIQUE INDEX uk_employee_no (employee_no);
```

数据库层面保证不重复。代码层捕获冲突后重试（默认 3 次），因为冲突概率极低（需两个 HR 同一秒批同一部门入职），重试后几乎必成功。

### 风险 3：定时任务幂等

**策略**：以「状态 + 日期」为条件，SQL 天然幂等。

```sql
UPDATE employee SET status = 5 WHERE status = 3 AND resign_date <= CURDATE();
```

UPDATE 是原子操作，第二次执行时 WHERE 条件已不满足，不会重复处理。提醒类任务可能产生重复通知，但不会破坏数据一致性。

### 风险 4：顺序门控影响存量

**策略**：门控仅对新产生的审批生效。存量数据若受影响，备选方案为门控做宽松校验（仅提示不阻断）。

### 风险 5：通知仅站内信

**策略**：已抽象接口，后续接 RocketMQ/邮件只需改配置，调用方零改动。本期在 proposal 中明确声明为 Non-Goal。

### 风险 6：转正辞退分支

**策略**：「不通过」默认置「待离职」并产生提醒通知 HR，是否自动发起离职审批由配置开关控制。

```java
hrms.regularization.reject-action = NOTIFY_HR  // 默认：通知HR手动处理
// 可选值：AUTO_RESIGNATION  // 自动发起离职审批
```

默认关闭自动联动的原因是：HR 可能想先和员工沟通再决定处理方式，一刀切自动触发反而限制操作空间。

## 八、Open Questions 处理策略

| 问题 | 是否阻塞执行 | 策略 |
|------|:--:|------|
| 入职二审判定规则 | ⚠️ 仅阻塞 task 2.8 | 核心链路 2.1-2.7 不依赖此规则，可先写死"全部走二审"，2.8 放到入职流程最后做 |
| 转正辞退联动 | ❌ 不阻塞 | 配置开关已兜底，默认通知 HR 手动处理 |
| 通知渠道 | ❌ 不阻塞 | 接口抽象已兜底，已声明 Non-Goal |
| 工号复用策略 | ⚠️ 仅阻塞 task 5.4 | 建议本期"只释放、不复用"，年份天然隔离了不同年度的工号，以后实际遇到再定复用策略 |

### 执行建议

约 50 个 task 可以立即开始执行，不受 Open Questions 影响。仅有 task 2.8（入职二审条件化）和 task 5.4（工号释放复用）需要后续拍板。
